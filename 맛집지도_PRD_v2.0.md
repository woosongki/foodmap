## 0. 맛집의 정의 (등록 기준)

**본 앱에서 "맛집"이란 무엇인가** — 이 정의는 모든 기능보다 우선합니다. 등록 여부가 모호할 때 이 기준으로 판단합니다.

### 0.1 등록 자격

| 구분 | 조건 | 표시 |
|------|------|------|
| **직접 방문한 곳** | 0.2의 판단 3축 중 **최소 1개 이상** 충족 | "방문완료" |
| **추천받은 곳** | 단이님이 신뢰하는 사람의 **강력 추천**이 있는 곳 | "위시리스트" (방문 후 정식 등록 여부 재판단) |

> **원칙**: 단순히 "유명한 곳", "리뷰가 많은 곳"은 등록 자격이 아님. 본인의 경험 또는 신뢰하는 사람의 추천만 등록.

### 0.2 판단 3축

다음 세 가지 중 **최소 하나 이상이 명확하게 충족**되어야 등록 가치가 있는 곳으로 봄.

| # | 기준 | 정의 |
|---|------|------|
| 1 | **맛** | 음식 자체의 완성도. 평범한 메뉴라도 수준이 분명히 다른 곳 |
| 2 | **재방문 의사** | 다음에 또 가고 싶은 마음이 드는 곳 (단순 "괜찮았다"가 아닌 "또 가야지") |
| 3 | **희소성·특별함** | 동네에 흔하지 않은 메뉴·공간·경험을 제공하는 곳 |

> 셋 다 평범하면 등록하지 않음. 하나만 압도적이어도 등록 가치 있음.

### 0.3 수준 태그 (등급 대체)

별점이나 등급 대신 **태그로 표시**. 각 식당에 0~여러 개 부여 가능.

| 태그 | 의미 |
|------|------|
| `#재방문` | 이미 두 번 이상 방문 또는 재방문 확정 의사 |
| `#감동` | 음식·공간·서비스에서 기억에 남는 인상이 있었던 곳 |
| `#일차` | 1차로 가기 좋은 곳 (식사 위주, 동선 시작점) |

> 추후 단이님 사용 패턴에 따라 태그는 자유 추가·변경 가능. 위 3개는 초기 권장값.

### 0.4 미등록 / 등록 해제 기준
- 방문 후 0.2의 3축 중 **어느 것도 충족하지 못한 경우** → 등록하지 않거나 삭제
- 위시리스트 상태로 6개월 이상 방치된 곳 → 재검토 (실제로 갈 의사가 있는지)

### 0.5 기본 레이어 (외부 공신력 데이터)

직접 등록한 식당 외에, **공신력 있는 외부 기준**을 기본 레이어로 제공. 등록 출발점이자 비교 기준으로 활용.

| 레이어 | 핀 라벨 | 아이콘 | 출처 | 범위 |
|--------|---------|--------|------|------|
| 내 맛집 | (없음) | ❤️ | 단이님 직접 등록 (§0.1~0.3) | 전국 |
| 백년가게 | **백년** | 🏛️ | 중소벤처기업부 백년가게 인증 | 주요 도시 (서울/부산/대구 등) |
| 리뷰 많은 곳 | **리뷰** | ⭐ | 네이버 리뷰 1만 건 이상 | 주요 도시 (서울/부산/대구 등) |

**표시 정책**
- 기본 상태: **OFF** (꺼진 상태로 시작) — 내 맛집에만 집중
- 토글 UI: 지도 상단에 `백년` `리뷰` 토글 버튼 → 각각 켜고 끄기 가능
- 동시에 켤 수도 있고, 따로 켤 수도 있음

**주요 도시 정의 (초기)**: 서울특별시, 부산광역시, 대구광역시, 인천광역시, 광주광역시, 대전광역시, 울산광역시
> 운영 중 추가/조정 가능

**확인 필요 사항 (실제 빌드 전 검증)**
1. **백년가게 데이터**: 중소벤처기업부 또는 소상공인시장진흥공단에서 운영하는 "백년가게" 제도의 공식 명단·좌표 데이터 확보 경로가 명확하지 않음. 공공데이터포털(data.go.kr) 또는 직접 크롤링 가능 여부 확인 필요
2. **네이버 리뷰 수**: 네이버 검색 API의 지역 검색 응답에 리뷰 수가 포함되어 제공되는지 미확인. 또한 "리뷰 1만 건 이상 식당"을 전수 조회할 수 있는 엔드포인트는 존재하지 않을 가능성이 높음 — 검색어 기반으로만 조회 가능할 수 있음
3. **법적·이용약관 검토**: 외부 데이터를 본인 앱에 표시하는 방식이 각 출처의 이용약관에 부합하는지 확인 필요

→ 위 사항이 해소되지 않을 경우, 차선책으로 **수동 큐레이션** (단이님이 신뢰하는 매체·블로그·언론 보도 기반 리스트를 수동 입력) 고려

## 1. 프로젝트 개요

### 1.1 배경
가족·지인과 공유 가능한 개인 맛집 지도가 필요함. 기존 네이버 MY플레이스나 카카오맵 즐겨찾기는 공유·관리·태깅의 자유도가 낮음. 자체 빌드를 통해 원하는 분류와 메모 체계를 구성하고, 모바일에서 외출 중에도 바로 활용 가능한 형태를 지향함.

### 1.2 목적
- 전국 단위 맛집 정보를 한 곳에 모아 시각적으로 확인
- 외출 중 "근처 어디 갈까" 의사결정을 빠르게 지원
- 가족·지인이 별도 로그인 없이 보기 가능, 편집은 본인만

### 1.3 사용자
| 구분 | 역할 | 접근 권한 |
|------|------|----------|
| 단이 (Owner) | 등록·수정·삭제 | 편집 가능 |
| 가족·지인 (Viewer) | 조회·길찾기 연결 | 읽기 전용 |

## 2. 핵심 기능 (MVP)

### 2.1 필수 기능 (Must-have)

| # | 기능 | 설명 |
|---|------|------|
| F1 | 식당 등록 | 네이버 검색 API로 식당 검색 → 결과에서 선택해 등록 (보조: 수동 입력) |
| F2 | 지도 핀 표시 | 네이버 지도 위에 등록된 식당을 핀으로 표시 |
| F3 | 상세 정보 | 핀 클릭 시 사진·메모·태그·방문여부 표시 |
| F4 | 방문/위시 구분 | 핀 색상으로 "방문완료" vs "위시리스트" 구분 (§0.1 기준) |
| F5 | 사진 관리 | **네이버 검색 이미지 URL만 저장 (식당당 최대 3장)**. 직접 업로드 없음. 네이버에서 원본 삭제 시 깨질 수 있음 |
| F6 | 수준 태그 + 자유 태그 | **수준 태그**(#재방문 #감동 #일차, §0.3) + **자유 태그**(#한식 #주차가능 등) 분리 입력 |
| F7 | 주변 검색 | 현 위치 기반으로 근처 등록 식당 표시 |
| F8 | 보기 전용 공유 | 단일 URL로 가족·지인이 로그인 없이 조회 가능. **편집 화면은 비밀번호 입력 필요** |
| F9 | 기본 레이어 토글 | `백년` `리뷰` 토글로 외부 공신력 데이터 표시 ON/OFF (§0.5) |

### 2.2 향후 고려 (Nice-to-have, MVP 이후)
- 동선·코스 짜기 (여러 식당을 묶어 코스로 저장)
- 리뷰 본문 작성·검색
- 외부 길찾기 앱 연동 (네이버 지도 앱으로 바로 이동)
- 통계 (가장 많이 방문한 지역, 태그별 분포 등)

## 3. 기술 사양

### 3.1 플랫폼
- **타겟**: 모바일 웹 우선 (반응형, 외출 중 사용)
- **데스크탑**: 동일 URL로 접근 가능, 핀 등록·편집 시 활용
- **앱 설치 없음**: 브라우저 기반 PWA 또는 일반 웹앱

### 3.2 기술 스택 (확정)

| 영역 | 선택 | 비고 |
|------|------|------|
| 프론트엔드 | HTML / JavaScript (단일 파일 또는 React) | 이전 팝업 캘린더 v5 방식 참고 가능 |
| 지도 | 네이버 지도 API (Maps JavaScript API v3) | 네이버 클라우드 플랫폼 계정 필요 |
| 검색 | 네이버 검색 API (지역 검색) | 별도 키 발급 필요 |
| 데이터 저장 | **Supabase** (PostgreSQL) | 식당 정보·태그·상태 저장 |
| 사진 저장 | **없음 (URL만 저장)** | 네이버 검색 이미지 URL을 DB에 텍스트로 저장 |
| 편집 인증 | **비밀번호 방식** | 편집 화면 진입 시 비밀번호 입력 |
| 호스팅 | **Vercel** | GitHub 연동 자동 배포 |
| 코드 저장소 | **GitHub** | 프라이빗 레포 권장 |

### 3.3 사전 준비사항 (Phase 1 진입 전 발급·생성 필요)

다음은 단이님이 직접 발급·생성하셔야 하는 항목입니다. **무료 사용량 한도는 시점에 따라 변동되므로 실제 신청 시점에 각 서비스 공식 문서를 확인해야 합니다.**

| # | 항목 | 발급처 / 생성 위치 | 비고 |
|---|------|----------------|------|
| P1 | 네이버 지도 API 키 | 네이버 클라우드 플랫폼 (ncloud.com) | Maps JavaScript API |
| P2 | 네이버 검색 API 키 | 네이버 개발자센터 (developers.naver.com) | 지역 검색 + 이미지 검색 |
| P3 | Supabase 프로젝트 | supabase.com | DB URL + Anon Key 발급 |
| P4 | GitHub 레포 | github.com | 프라이빗 권장 |
| P5 | Vercel 계정 | vercel.com | GitHub 연동 |
| P6 | 도메인 (선택) | - | 미설정 시 vercel.app 기본 도메인 사용 |
| P7 | 편집 비밀번호 | 단이님이 직접 정하기 | 환경변수로 관리 (Vercel에 등록) |

### 3.4 데이터 모델 (Supabase 테이블 초안)

```sql
-- 식당 메인 테이블
CREATE TABLE restaurants (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  name         text NOT NULL,                    -- 식당명
  address      text NOT NULL,                    -- 주소
  lat          double precision NOT NULL,        -- 위도
  lng          double precision NOT NULL,        -- 경도
  naver_url    text,                             -- 네이버 플레이스 URL
  status       text NOT NULL,                    -- 'visited' | 'wishlist'
  source       text NOT NULL DEFAULT 'self',     -- 'self' | 'recommendation'
  recommender  text,                             -- 추천인 (source='recommendation'인 경우)
  memo         text,                             -- 자유 메모
  -- 판단 3축 (status='visited'인 경우 최소 1개 필수)
  axis_taste     boolean DEFAULT false,          -- 맛
  axis_revisit   boolean DEFAULT false,          -- 재방문 의사
  axis_unique    boolean DEFAULT false,          -- 희소성·특별함
  created_at   timestamptz DEFAULT now(),
  updated_at   timestamptz DEFAULT now()
);

-- 태그 (수준 태그 + 자유 태그)
CREATE TABLE tags (
  restaurant_id uuid REFERENCES restaurants(id) ON DELETE CASCADE,
  tag           text NOT NULL,
  tag_type      text NOT NULL,  -- 'level' (재방문/감동/일차) | 'free' (자유 태그)
  PRIMARY KEY (restaurant_id, tag)
);

-- 사진 URL (네이버 이미지 URL만 저장, 최대 3장)
CREATE TABLE photos (
  id             uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  restaurant_id  uuid REFERENCES restaurants(id) ON DELETE CASCADE,
  image_url      text NOT NULL,
  display_order  smallint NOT NULL  -- 1, 2, 3
);

-- 기본 레이어 (백년가게, 리뷰 1만+) - 별도 테이블로 관리
CREATE TABLE external_landmarks (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  name         text NOT NULL,
  address      text NOT NULL,
  lat          double precision NOT NULL,
  lng          double precision NOT NULL,
  layer_type   text NOT NULL,  -- 'baeknyeon' (백년가게) | 'review' (리뷰 1만+)
  city         text NOT NULL,  -- '서울' | '부산' | ...
  source_url   text,           -- 원본 데이터 출처
  created_at   timestamptz DEFAULT now()
);
```

> 위 스키마는 v2.0 초안. Phase 1 진입 시 Supabase에 적용 후 운영 중 조정 가능.

## 5. UI / UX 개요

### 5.1 화면 구성 (모바일 기준)

```
┌────────────────────────┐
│  [검색바: 식당명/지역]   │
├────────────────────────┤
│  [백년 OFF] [리뷰 OFF]  │ ← 기본 레이어 토글
├────────────────────────┤
│                        │
│      [네이버 지도]      │
│   ❤️ 내 맛집           │
│   🏛️ 백년 (토글 시)    │
│   ⭐ 리뷰 (토글 시)    │
│                        │
├────────────────────────┤
│ [태그필터] [내주변] [+] │
└────────────────────────┘
```

### 5.2 주요 인터랙션
1. **등록**:
   - `+` 버튼 → 네이버 검색 → 결과 선택
   - **상태 선택**: 방문완료 / 위시리스트 (§0.1)
   - 방문완료 선택 시 → **판단 3축 체크박스** (맛 / 재방문 의사 / 희소성, §0.2) — 최소 1개 필수
   - **수준 태그** (재방문·감동·일차, §0.3) + **자유 태그** 입력
   - 메모·사진 추가 → 저장
2. **조회**: 핀 클릭 → 하단 시트(bottom sheet)에 상세 정보 (판단 3축 + 태그 + 메모 + 사진)
3. **필터**: 태그 칩 선택 → 해당 태그 식당만 표시 (수준 태그 / 자유 태그 분리 필터)
4. **내 주변**: 현 위치 기준 반경 N km 내 등록 식당만 표시
5. **공유**: 공유 URL 복사 → 카톡·문자로 전달

## 8. 본 PRD의 한계 및 리스크

v2.0은 의사결정이 모두 확정된 빌드 진입 가능 버전이지만, 다음 사항은 여전히 검증·주의가 필요합니다.

### 8.1 외부 의존성 리스크
1. **네이버 이미지 URL 깨짐**: 네이버 측에서 원본 이미지 삭제 시 식당 사진이 깨짐. 시간이 지나면 일부 식당의 사진이 빈 상태가 될 수 있음 — **감수하기로 결정**
2. **네이버 API 정책 변경**: 무료 한도 축소·유료 전환 가능성 (시점에 따라 변동)
3. **Supabase 무료 프로젝트 일시 정지 정책**: 일정 기간 미사용 시 정지될 수 있음 — 정기 접속 필요할 수 있음

### 8.2 보안·개인정보
1. **비밀번호 방식의 한계**: 환경변수에 저장하지만, 노출 시 누구나 편집 가능. 강력한 비밀번호 필수 + 정기 교체 권장
2. **공개 URL의 위치 정보 노출**: 자택·자녀 동선 근처 핀이 가족 외 지인에게도 노출됨 — 별도 비공개 핀 기능은 v2.0 범위 밖, 필요 시 v3에서 검토

### 8.3 확인 필요 사항 (§4.7 참고)
Q1~Q6은 빌드 진행 중 병렬로 검증 진행

### 8.4 본 PRD 작성의 한계
- 본 문서의 무료 한도·약관 관련 기술은 **단정이 아니며 검증 필요**
